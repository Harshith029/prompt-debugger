# ADR-0008: Vendor core/ into the Claude Code plugin

- **Status:** Accepted
- **Date:** 2026-07-07

## Context

The host-neutral design (ADR-0003) keeps contracts and knowledge in a repo-root `core/` tree. But a Claude Code plugin is installed as a self-contained directory; at runtime a skill cannot reach "up" to a repository root to read `core/`. The adapter needs the contracts and knowledge locally.

## Decision

The Claude Code adapter **vendors** a copy of `core/` inside `adapters/claude-code/core/`. Repo-root `core/` remains the **single source of truth**; the vendored copy is generated by `tools/sync_plugin.py` and verified by `tools/check_plugin_sync.py`, which fails CI on any drift. The vendored copy is marked generated and excluded from linting.

## Alternatives considered

- **Symlink the plugin's core/ to repo-root core/.** Rejected: symlinks do not survive plugin packaging/installation across platforms (Windows especially), so the installed plugin would have a dangling link.
- **Duplicate content by hand.** Rejected: guaranteed to drift. Generation + a drift check keeps one source of truth.
- **Load core/ from a package data directory at runtime.** Rejected for v1: the skills read Markdown companions directly from the plugin tree; a packaged-data indirection adds complexity the adapter does not need yet.

## Trade-offs

- The repository contains a generated duplicate of `core/`. Mitigated: it is clearly stamped generated, excluded from lint, and CI enforces that it is byte-identical to the source.
- Contributors must run `sync_plugin.py` after editing `core/` (or CI reminds them).

## Consequences

- The installed plugin is self-contained and needs no repository.
- Repo-root `core/` stays authoritative; drift is impossible to merge (CI gate).
- The same vendoring pattern is available to any future adapter that needs self-containment.
