This is the permanent engineering record for Milestone M0. It documents what the milestone set out to do, what was built, the decisions that shaped it, how it was verified, and the state in which it is frozen and handed to M1.

## 1. Milestone objective

Establish a production-quality engineering foundation for `prompt-debugger` — the contracts, knowledge structure, tooling, tests, documentation, and CI on which all later functional milestones build — **before** writing any analyzer or rewrite logic. The guiding principle: get the interfaces, invariants, and guarantees right first, so feature work has a stable, reviewable base and cannot silently violate the project's privacy and honesty commitments.

## 2. Completed work

- **Host-neutral core contracts (8):** Prompt IR, Report JSON, Rewrite Report, Observable Event, Storage (history record + config), Knowledge Engine file formats, Plugin/Adapter API, and Prompt Tree — each a JSON schema (restricted, recursion-free subset) plus a prose `CONTRACT.md`.
- **Knowledge Engine:** versioned data packs separating prompt-engineering knowledge from code — a provider-neutral `common` rubric pack (R1–R10) and an `anthropic` provider pack (dated claim registry, techniques T1–T10, observable-event taxonomy, pattern library with authored bodies).
- **Contract invariants:** a catalogue (`docs/CONTRACT-INVARIANTS.md`) of every rule the schema subset cannot express, each with an enforcement point and, where practical, an executable test — including the persisted-record redaction invariant (PR-1).
- **Claude Code adapter scaffold:** plugin/marketplace/adapter manifests, four skill skeletons (`analyze`, `rewrite`, `history`, `pd`) with a corrected tool-permission model, and a CI-verified vendored copy of `core/`.
- **Benchmark corpus:** nine categories with a schema-validated runner, wired into CI.
- **Semantic-evaluation protocols:** meaning preservation, red-team rewrite, and rubric calibration, as documented stubs for the release gate.
- **Tooling:** schema/subset validation, AST import-policy enforcement, relative-link checking, plugin vendor sync, version consistency, and release-version verification.
- **Documentation:** architecture, eight ADRs, data flow, glossary, threat model, privacy model, roadmap, compatibility policy, ethics/use policy, contract invariants, and the Prompt Tree design.
- **CI/CD:** a three-OS × two-Python matrix gate and a release workflow (running the full gate plus version/tag checks) with checksums and build-provenance attestations.

## 4. Verification summary

- Independent review at each stabilization pass (M0.1, M0.1.1); every accepted finding was resolved before closure.
- The privacy invariant PR-1 is enforced by an executable test with both a compliant fixture and a leak fixture that proves the guard is non-vacuous.
- "No telemetry / no network" is enforced behaviorally (AST import allowlist + socket-blocking test harness), not merely documented.
- Composite validation is exercised end-to-end (report → IR / event / rewrite, and embedded in a history record) with populated fixtures.

## 7. Lessons learned

- **Contracts-first pays off early.** Freezing schemas and invariants before code surfaced the P0 privacy defect (a secret scrubbed from the prompt could survive in the embedded report) at the contract level, where it was cheap to fix, rather than after an implementation existed.
- **Make guarantees executable.** Turning the privacy and rewrite invariants into tests with negative fixtures converted "documented promises" into enforced properties and removed a class of silent regressions.
- **Documentation drifts fastest.** The stabilization passes repeatedly caught docs describing not-yet-built modules in the present tense; the fix was to tie every forward-looking statement to a milestone label and audit for it.
- **Host-neutrality is a discipline, not a one-time choice.** Provider-specific values (the event `surface` enum) leaked into the "neutral" core and had to be pushed back into the provider pack — worth re-auditing at each milestone.

## 8. Known limitations (intentional at M0)

- No analyzer, rewriter, storage, redaction, verification, or rendering implementation exists — those are M2–M4. The `history` skill's launcher reports "not implemented" by design.
- Knowledge claims are `status: "recorded"` and techniques/taxonomy are `draft`; verification against live sources is M1.
- Many invariants (IR/report/storage/tree) have reference tests but defer runtime enforcement to their M2 components.
- The benchmark corpus has one representative case per category — sufficient for integrity checks, not yet a measurement instrument.

## 9. Transition to M1

M0 is **frozen**: architecture, contracts, schemas, Knowledge Engine structure, and repository layout do not change except to fix an objective defect. M1 begins the first knowledge-facing work — verifying every claim against its live source, authoring the misuse policy / rewrite contract / notices in the `common` pack, expanding the taxonomy prose, and resolving the open `clm-consumer-surface-fallback` question — without touching analyzer or rewrite logic. The M1 product specification is [`specs/M1.md`](../../specs/M1.md).
