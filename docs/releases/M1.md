This is the permanent engineering record for Milestone M1. It documents what the milestone set out to do, what was delivered, how it was verified, and the state in which the Knowledge Engine content is frozen as the verified baseline for M2. The governing specification is [`specs/M1.md`](../../specs/M1.md).

## 1. Milestone objective

Convert the M0 seed knowledge into a **verified, authored, testable corpus**: every factual claim about provider behavior checked against its live public source; the misuse policy, rewrite-transformation rules, and fixed notices authored as versioned data; the observable-event taxonomy's honesty boundaries completed; the pattern library populated; and the whole corpus version-labelled — because the product's value proposition (sourced accuracy and a specified, testable no-bypass policy) cannot rest on unverified claims or an unwritten policy.

## 3. Engineering summary

M1 introduced **content and its executable integrity floor, not runtime code**. The corpus work was accompanied by: three additive knowledge schemas (`misuse-policy`, `rewrite-policy`, `notices`) under `core/contracts/knowledge/` taken as an approved frozen-baseline exception; one optional field added in-version to the Report contract (`report.knowledge.policy_version`, per its additive compatibility rule) so reports can pin the policy that produced them; invariants KN-7–KN-10, PL-1–PL-8, and EV-4 catalogued and made executable (30 new tests, 54 → 84); the governance process for policy evolution (`docs/process/policy-review.md`); ADR-0009 (truncation observables deferred); and the formatter pinned exactly (`ruff==0.15.21`) after a cross-environment verification discrepancy was root-caused to the unpinned version range.

## 4. Architecture delivered

No structural change — M0's architecture is intact. Within it, M1 delivered the **policy layer as data**: layered policy design ([`docs/design/policy-architecture.md`](../design/policy-architecture.md)) and field-by-field schema design ([`docs/design/policy-schemas.md`](../design/policy-schemas.md)) with stable permanent registry ids (`MISUSE-###`, `RW-###`, `NOTICE-###`), independent `schema_version`/`policy_version` axes, per-schema compatibility rules, and maintainer rationale on every entry. The policy system is purely declarative: closed enums and prose, no branching fields, no rule engine — the M2+ engine interprets it and never executes it.

## 5. Contracts used

The 8 frozen M0 contracts, unchanged except: the Report contract's additive optional `knowledge.policy_version`; the Rewrite Report contract's notices pointer corrected to the JSON source of truth; and the 3 additive knowledge schemas above. The `policy_version` pattern was calendar-tightened (`(0[1-9]|1[0-2])` months) under a recorded pre-1.0 exception — no valid instance existed in the rejected space.

## 6. Knowledge corpus delivered (snapshot `2026.07-m1`)

- **Claim registry:** 12 claims, all `verified` with URLs, retrieval dates, and `last_verified` dates, spanning the platform API docs, the stop-reasons and errors references, the Fable 5 prompting guide, the prompting best-practices corpus, the Claude Code model-configuration docs, the Claude Help Center, and the official Cookbook. Includes one verified *negative* (`clm-consumer-surface-fallback`, page-scoped).
- **Techniques:** T1–T10, each claim-cited.
- **Rubric:** R1–R10 (provider-neutral, in `common`).
- **Policy layer:** see §7. **Taxonomy:** see §8. **Patterns:** see §9.
- Provenance is end-to-end executable: KN-1 (statements cite claims), KN-2 (active cites verified), KN-7 (events cite only verified; lifecycle + provenance fields checked), KN-8 (no orphaned verified claims), KN-10 (snapshot labels agree).

## 7. Policy layer delivered

`core/knowledge/packs/common/`: `misuse-policy.json` (legitimate/ambiguous/prohibited classes mapped to the Rewrite-Report gate space; the 5-step neutral procedure as an ordered prose checklist — no `on_yes`/`on_no` branching; ask-don't-guess elicitation; fixed decline templates), `rewrite-policy.json` (RW-001–009 allowed transformations with technique provenance; RW-050–055 prohibited intent-laundering checklist; RG-1–RG-8 guarantees each marked `hard`/`judged`; notice-attachment rules), `notices.json` (the single source of truth for all fixed notice wording). Each has a drift-checked prose companion; JSON is canonical for all fixed user-facing wording (PL-7). Frozen after independent review; evolution is governed by [`docs/process/policy-review.md`](../process/policy-review.md).

## 8. Taxonomy delivered

Six observable-event entries with production `can_conclude`/`cannot_conclude` boundaries, each statement traceable to a verified claim. Key properties: the HTTP-200-not-error framing of refusals; the truncation-vs-error separation (stop conditions are not HTTP errors — ADR-0009 defers a dedicated truncation kind until M2's contract review); per-surface citation discipline for model switches (Claude Code and the consumer apps have documented behavior; undocumented surfaces remain the user's report; one surface's documentation is never borrowed for another); and the deterministic observation-channel selection rule — API-native kinds list exactly the `api` surface, rendered kinds never do, so no observation matches two entries (EV-4, executable).

## 9. Pattern library delivered

Ten before/after patterns, one per rubric dimension (the documented completeness definition — patterns exist to show the fix for a *finding*). Every After demonstrates only rewrite-policy-permitted transformations: information the user holds but the prompt lacks appears as explicit angle-bracket slots or questions, never as invented dates, metrics, audiences, technologies, or domain facts; requirement strength is preserved exactly. The objectively checkable slice is CI-enforced (an After's numeric/file-like literals must appear in its Before); prose-level compliance is enforced by the documented rule plus review. T5 ("give Claude a role") has no dedicated pattern because no rubric dimension maps to it.

## 10. Version bookkeeping and the promotion decision

The corpus is labelled **`2026.07-m1`** in every snapshot field (manifest, both packs, rubric, taxonomy, three policy files) and companion header, with agreement CI-enforced (KN-10). The `-m1` tag is deliberate: per-entry `status` remains `"draft"` across techniques, taxonomy, rubric, and patterns. The spec's FR-2 is permissive ("whose supporting claims are all `verified` **may** become `active`"), and its acceptance criterion is the constraint — no active entry citing a non-verified claim — which holds and is executable. **Bulk promotion was deliberately not performed in M1**: no consumer binds to entry statuses until the M2 analyzer exists, and flipping statuses without a consumer would be an unobservable change. Every claim is `verified`, so promotion is unblocked; when M2 exercises it, the status change is a content change requiring a snapshot bump under KN-10. The independent M1 approval covered the repository in this state.

- Independent review gated every FR; every verified finding was resolved in a numbered stabilization pass (FR-3.1/3.2, FR-4.1–4.4, FR-5.1) before proceeding — including two review corrections of this project's own work product: the consumer-surface overstatement (fixed by surveying and citing the real product documentation) and rewrite-policy violations in the pattern examples (fixed with the placeholder convention).

## 14. Lessons learned

- **Examples are policy too.** The pattern library's own Before/After examples initially violated the rewrite guarantees they demonstrate (invented context, narrowed scope). Content that *teaches* a rule must satisfy the rule, and the objectively checkable slice of that must be a test, not a review note.
- **A page-scoped negative is not a global negative.** "This page doesn't document X" hardened into "X is not publicly documented" and was falsified by a wider survey. Claims must be scoped to exactly their evidence, and the registry now records where a behavior *is* documented alongside where it is not.
- **Unpinned formatters make the format gate non-deterministic.** Two conforming environments disagreed on the same tree. The gate is only a gate if its tool is pinned.
- **Vocabulary drift breaks enforcement silently.** The KN-2 test checked for a claim status (`active`) that has never existed in the lifecycle, so the invariant was unenforceable from the day it was written. Cross-checking test vocabulary against schema enums is itself now a test.
- **Deterministic selection needs disjoint definitions.** Two taxonomy entries could match one observation until the observation-channel rule made their surface sets provably disjoint — ambiguity is removed by construction, not by tie-breaking.

## 15. Known limitations (intentional at M1)

- No analyzer, rewriter, storage, redaction, verification, or rendering implementation exists — those are M2–M4. The `history` launcher still reports not-implemented by design.
- All knowledge entries remain `status: "draft"`; promotion is deferred to M2 as recorded in §10.
- Truncation stop conditions have no dedicated taxonomy kind (ADR-0009); M2's contract review must revisit.
- The pattern-example fabrication check covers the objective slice (numeric/file-like literals); prose-level fabrication remains review-enforced.
- `mypy`/`pytest`/`jsonschema` remain range-pinned (only ruff is exact); the same determinism treatment is a candidate at the next infrastructure pass.
- The benchmark corpus remains one case per category — an integrity check, not yet a measurement instrument.

## 16. Transition to M2

M1 is **frozen**: the verified claim registry, policy layer, taxonomy, pattern library, and their schemas change only to fix objective defects (policy content additionally follows [`docs/process/policy-review.md`](../process/policy-review.md)). M2 builds the core library — storage, the schema-subset validator, the evidence verifier, redaction, sanitization, rendering, CLI — implementing as code the invariants this milestone recorded as data and reference tests. Explicit M2 obligations inherited from M1: revisit ADR-0009 (truncation kind), perform status promotion when the analyzer binds to statuses (with the KN-10 snapshot bump), and add the differential `schema.py`-vs-`jsonschema` validation once `schema.py` exists.
