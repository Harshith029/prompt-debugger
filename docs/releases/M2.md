This is the permanent engineering record for Milestone M2. It documents what the milestone set out to do, what was delivered, how it was verified, and the state in which the core library is frozen as the deterministic base on which M3 builds the analyzer. The governing specification is [`specs/M2.md`](../../specs/M2.md).

## 1. Milestone objective

Implement as code what M0 recorded as contracts and M1 recorded as verified knowledge: a stdlib-only runtime library that validates, verifies, redacts, sanitizes, stores, renders, and exposes the corpus — so that nothing user-facing rests on unchecked structure. Until this milestone, the privacy invariant PR-1 was enforced only against fixtures, evidence quotes were verified only by reference tests, and the `history` skill had no CLI to dispatch to.

## 3. Engineering summary

M2 added **runtime code and nothing else**: eight new stdlib-only modules (2,696 lines) and eleven new test modules, taking the suite from 84 to 264 tests. No runtime dependency was introduced; the AST import allowlist and the socket-block harness pass unchanged (ADR-0006). The work was overwhelmingly defensive: the large majority of the verified review findings resolved across the milestone were fail-closed and privacy defects — post-redaction PR-1 enforcement, incomplete-write detection, symlink races, untrusted-record reads, terminal-escape injection through four distinct output paths, and Markdown structure injection.

## 4. Architecture delivered

No structural change — M0's architecture is intact and now populated. The `src/prompt_debugger/` box the architecture reserved for M2 is filled: `schema`, `verify`, `redact`, `sanitize`, `store`, `render`, `knowledge`, `cli`, alongside the M0 `paths` and version constants. The layering held: the CLI wraps the library and adds no behavior of its own; `render` projects reports and never authors them; `verify` returns violations and never repairs them; the declarative policy layer stays data that nothing in M2 executes.

## 5. Contracts used

The 8 frozen M0 contracts, **unchanged**. M2's single carved-out exception (an additive Observable Event `kind` member, had FR-9 decided to add one) went unused. The only schema edit was the `size_warn_records` default in `config.schema.json` — a value the schema itself had documented as "provisional until M2 benchmarks set a measured value" (§10). `CONTRACT_VERSIONS` and every schema `const` version remain at 1.

## 6. Library delivered

Eight modules behind one rule — everything checkable is checked by code, and every refusal is specific:

- **Validation** (`schema`): subset conformance plus draft-2020-12 instance semantics, with composite validation for reports and history records; a schema outside the subset is a fail-closed error, not a silent pass.
- **Verification** (`verify`): the relationships schemas cannot express, returned as machine-readable violations against a caller-supplied reference prompt and taxonomy id set (EV-2 fail-closed on an empty set).
- **Privacy** (`redact`, `sanitize`): typed-placeholder redaction and terminal/CSV output safety, both deterministic and idempotent.
- **Persistence** (`store`): the full operation set with atomic appends under an advisory lock, quarantine, migration, and archival.
- **Projection** (`render`) and **access** (`knowledge`, `cli`).

## 7. Storage and privacy delivered

PR-1 through PR-6 are enforced at the real write path, not against fixtures. A `raw: false` record is redacted in **every** field including the embedded report, and the record must additionally be a fixed point of the committed redactor and satisfy the post-redaction substring relations — a save that cannot satisfy this is refused, and `doctor` quarantines any stored record that violates it. Raw persistence is rare, loud, and reversible: refused without explicit per-save confirmation, visibly flagged in listings, and revertible via `strip-raw`. Ids (PR-2), store-local HMAC fingerprints excluded from default exports (PR-3), atomic appends with tolerated partial tails (PR-4), symlink refusal and containment checked immediately before every filesystem write (PR-5), and migration accepting every shipped `record_version` (PR-6) are all executable.

## 8. Knowledge promotion decision (FR-8)

The spec's criterion was applied mechanically per entry — promote where **every cited claim is `verified`** (the KN-2 posture). Since all 12 claims have been `verified` since M1, this promoted **all 10 techniques (T1–T10)** and **all 6 taxonomy entries** to `active`. One interpretive point is recorded rather than silently absorbed: `evt-none` cites zero claims by design (kind `none` asserts no provider behavior — the KN-1 carve-out), so it satisfies the criterion vacuously and was promoted with the rest. **Rubric dimensions and patterns remain `draft`**: they carry `status` fields but no claim-provenance relation, so no repository artifact grounds a promotion criterion for them; defining one stays open choice O7, an explicit governance decision. The content change bumped the snapshot `2026.07-m1` → `2026.07-m2` coordinately across every label and companion header (KN-10), and the existing KN-2/KN-7/KN-10 integrity tests pass unmodified on the promoted corpus.

## 9. ADR-0009 outcome (FR-9)

**Re-deferred**, recorded in [ADR-0010](../adr/0010-truncation-kind-re-deferred.md), which supersedes ADR-0009. The criterion ADR-0009 set — "the first milestone where the analyzer needs to *classify* truncation" — is unmet: M2 built the deterministic library, not the analyzer. The member-granularity question (one `stop_condition` kind vs. per-stop-reason kinds over the seven documented values) remains undecidable without real classification needs. ADR-0010 also corrects ADR-0009's "no migration debt" framing against the Observable Event contract's own compatibility rule: adding a `kind` member is a version bump, so an M2 addition would have spent a rare v1→v2 evolution on a kind no component consumes. Truncation reports continue through the contract-mandated `unknown` honesty path. The obligation to revisit is carried forward to the milestone that authors the analyzer; a regression test pins the enum so the kind cannot be added silently.

## 10. Performance measurements (FR-11)

Measured with `benchmarks/perf/harness.py`; full data in [`benchmarks/perf/RESULTS.md`](../../benchmarks/perf/RESULTS.md). Every read path validates each record on read (the fail-closed S10 posture), so the scan is validation-bound.

| records | `list` (median of 5) |
|--------:|---------------------:|
|   1,000 |              621.5 ms |
|   2,000 |            1,280.2 ms |
|   5,000 |            3,133.0 ms |
|  10,000 |            6,378.7 ms |

The small-size region is linear at **≈ 0.63 ms/record**. At large sizes the scan degrades super-linearly with high run-to-run variance; two independent single-repeat runs are recorded as observed, without averaging — 50k `list` at 132.4 s and 66.4 s, 100k `list` at 180.6 s and 188.4 s. A naive linear projection would have predicted ~32 s and ~63 s, so the measured cost is materially worse than extrapolation and the measured figures replace it.

**Size-warning threshold.** `size_warn_records` was a guessed `50000`; it is now the measured **`1000`** — chosen conservatively below the modeled ~1,600-record one-second interactive-responsiveness crossover (a 1,000-record read measures ≈ 0.62 s). The threshold is wired into the CLI: `list` suggests `archive` at or above it. The dominant cost is per-record schema (re)loading inside the frozen validator; that optimization is deliberately **not** taken here and is recorded as forward work (§15).

## 11. Verification summary

- Independent review gated every FR; every verified finding was resolved before the next FR began, several through multi-round sequences (FR-1 ×4 review rounds, FR-5 ×3, FR-10 ×4, FR-11 ×5).
- Two review corrections overturned this project's own reasoning: that the restricted subset retains draft-2020-12 meta-schema constraints for the keywords it keeps, and that `json.dumps(ensure_ascii=False)` escapes only C0 — leaving C1 controls, including the 8-bit CSI introducer, to reach a terminal raw.
- The measurement corrected a documented assumption: the architecture's "single-digit milliseconds at realistic sizes" claim does not hold for the validation-bound read path, and the guessed 50,000-record threshold implied a ~31-second scan.
- Final state: every acceptance criterion in [`specs/M2.md`](../../specs/M2.md) verified against the repository at close-out, with PL-6 excluded exactly as the specification directs.

## 14. Lessons learned

- **Fail-closed has to be proven per path, not per module.** PR-1 was enforced at `append` but not after redaction, not in `doctor`, and not on the read paths. Each gap was a separate review round; an invariant is only as strong as its least-guarded entry point.
- **One output rule beats four.** Terminal-escape safety was fixed four times — stderr, CLI-owned stdout, JSON values, then argparse's own output — because each fix addressed a path rather than the boundary. The durable answer was to route every print surface through the single sanitizer.
- **Escaping is not sanitization.** `json.dumps` escapes C0 but, with `ensure_ascii=False`, emits C1 raw. A mechanism that *looks* like it neutralizes control characters must be checked against the actual byte ranges.
- **Extrapolation is not measurement.** The 100k figure projected from a clean linear fit was wrong by 3×; the requirement said measure, and only measuring found the super-linear degradation.
- **Documentation drifts silently while code moves.** Status text, module inventories, testing tables, and threshold prose each had to be corrected after the fact; a milestone that changes the repository's shape must budget for re-reading what the repository claims about itself.

## 15. Known limitations (intentional at M2)

- No analyzer, rewriter, or skill wiring exists — those are M3–M4. The `history` launcher still reports not-implemented by design, and no skill file changed in M2.
- PL-6 ("rationale never influences a decision") remains unenforced: M2 builds no policy decision engine to instrument. Its catalogue entry moves at the milestone that introduces policy execution.
- Rubric dimensions and patterns remain `draft`; their promotion criterion is open choice O7, a governance decision.
- Truncation stop conditions still have no dedicated taxonomy kind (ADR-0010); the revisit obligation passes to the analyzer milestone.
- Read scans re-load the contract schemas per record, which is why the measured threshold is low. Caching them is a real optimization, but it edits the frozen FR-1 validator and is therefore deferred to an approved, separately reviewed pass.
- `compare` produces the unified diff only; a rubric-score delta awaits an approved contract evolution that defines scoring.
- `mypy`/`pytest`/`jsonschema` remain range-pinned (only ruff is exact) — the same determinism treatment is still a candidate.

## 16. Transition to M3

M2 is **frozen**: the library, its tests, and the measured threshold change only to fix objective defects. M3 authors the `analyze` and `rewrite` skill behavior against this substrate — schema-validated contracts at runtime, machine-verified evidence, privacy-enforcing storage behind a CLI, deterministic rendering, and a promoted, versioned knowledge corpus behind read-only accessors. It is the first milestone in which model judgment enters the system, constrained on every side by what M0 specified, M1 verified, and M2 enforced. Explicit obligations inherited from M2: revisit ADR-0010 (truncation kind) when classification needs are real, decide open choice O7 (rubric/pattern promotion), and enforce PL-6 once a policy decision path exists.
