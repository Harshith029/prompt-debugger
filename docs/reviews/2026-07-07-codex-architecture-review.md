## Finding dispositions

| # | Finding (severity) | Disposition | Resolution in v0.2 | Rationale / trade-offs |
|---|---|---|---|---|
| F1 | No host-neutral core; everything depends on skill prompts (Critical) | **PA** | New layering: `core/` (versioned, host-neutral contracts: schemas, event taxonomy, rubric, rewrite/misuse policy, claim registry) + `src/prompt_debugger/` (stdlib library: storage, redaction, rendering, validation, sanitization) + skills as thin adapters. §4, ADR-0001. | Accepted: contracts and deterministic logic must be host-neutral so tests, storage, and future adapters don't depend on prose. **Partial**: we do *not* ship additional host runtimes (MCP/standalone CLI UX) in v1. The analysis/rewrite *judgment* is necessarily an LLM function; host-neutrality for that layer means neutral contracts + a portable methodology (Agent Skills open standard), not a second runtime. Building unproven adapters before the primary UX is validated is speculative generality; the contracts make them cheap later, which is the property the finding actually protects. |
| F2 | Markdown treated as the analysis engine; reports not machine-readable (High) | **A** | Dual-artifact output contract: every analysis produces a **Report JSON** (validated against `core/schemas/report.schema.json`) as the system of record, plus rendered Markdown as the human projection. Persistence and comparison operate only on Report JSON. §6.4, ADR-0002. | Trade-off: the model must emit structured JSON reliably; mitigated by schema validation at save time with a documented repair loop (validate → fix → revalidate), and by machine-verifiable evidence quotes (F3). |
| F3 | No Prompt IR; findings lack spans/structure (High) | **A** | Pragmatic **Prompt IR**: model segments the prompt into typed segments (role, context, task, constraint, example, output_spec, data, meta, other). Evidence is **verbatim substrings**, not character offsets — every quote is machine-verified against the source prompt by `validate.py`, giving a deterministic integrity check on LLM output. §6.3. | Verbatim spans chosen over offsets deliberately: LLMs are unreliable at arithmetic offsets but reliable at quoting; substring verification makes hallucinated evidence detectable. Non-exhaustive segmentation allowed (`unsegmented_remainder: true`) to avoid brittle full-tiling requirements. |
| F4 | `allowed-tools` mistaken for a restriction (Critical) | **A** | Threat model corrected. `analyze`/`rewrite` now set `disallowed-tools: Write Edit NotebookEdit Bash` (removes tools from the pool while active). `allowed-tools` retained only for its true purpose (pre-approval of history scripts). A documented **permission profile** ships in INSTALL.md, and a CI contract test fails if frontmatter drifts from the documented profile. §6.1–6.2, §7 S2/S13. | The v0.1 text did note the pre-approval semantics but the threat model still leaned on it; the reviewer is right that this was an unsound mitigation. |
| F5 | JSON Schema conflicts with stdlib-only; `jsonc` isn't JSON (High) | **A** | Schemas authored as standard **JSON Schema (draft 2020-12) restricted to a defined subset** (types, required, enum, properties/items, additionalProperties, pattern, min/max). Runtime validation via a small in-repo subset validator (`schema.py`, unit-tested). CI runs **differential validation**: the real `jsonschema` package (dev-only dependency) validates the same fixtures and must agree with our validator. The v0.1 `jsonc` example replaced with valid JSON + prose annotations. §6.7, ADR-0003. | Trade-off: we own ~150 lines of validator code vs. taking a runtime dependency. Differential testing in CI bounds the correctness risk; runtime supply chain stays empty, which was the point of stdlib-only. |
| F6 | Source claims too absolute; docs are living (High) | **A** | "Complete taxonomy" language removed. `core/sources/claims.yaml` is a **claim registry**: claim ID, exact claim text, URL, retrieval date, verification status, last-verified date. Taxonomy carries `taxonomy_version`; every report records the taxonomy/rubric versions used. Event classification includes a first-class **`unknown` event kind** with graceful, honest handling. Quarterly re-verification is a standing process (issue template + checklist). §2, §6.5, §13. | |
| F7 | Consumer-surface fallback claim unsupported (High) | **A** | Claim reframed. The *user-visible model-switch notice* is treated as **observed evidence reported by the user** (first-class input), while *citations* are restricted to documented API mechanisms (`stop_reason: refusal`, `stop_details`, fallback blocks). Whether a given surface implements fallback a particular way is stated as not publicly specified unless a public source is found — verifying the refusals-and-fallback page for consumer-surface language is an explicit M1 task, recorded in the claim registry either way. §2.1. | This preserves the flagship use case (Fable 5 flag → visible Opus 4.8 switch, confirmed by the maintainer) without asserting undocumented internals: the switch the user *saw* is a fact; the mechanism *citation* stays within documented surface. |
| F8 | Legitimacy gate vague (Critical) | **A** | Gate specified as a **decision procedure + transformation rules**, not category examples: dual-reading construction (most-plausible-legitimate vs most-plausible-harmful), explicit allowed/prohibited transformation lists (prohibits intent-laundering: fabricated context, fictional reframing, semantic dilution, salami-slicing, encoding), ask-don't-guess rule for missing purpose, refusal templates, and gate outcome recorded in Report JSON. Adversarial eval suite required at M3. `core/policy/` + §6.6. | Key invariant: **a rewrite may make intent more explicit, never less** — and may only use context the user actually provided. |
| F9 | Raw hashes are dictionary-attackable and leak via exports (High) | **A** | Prompt/rewrite fingerprints are **HMAC-SHA256 with a per-store random 256-bit salt** generated at store init and held in the store directory. Fingerprints serve only local revision-matching and are **excluded from all exports by default**. §6.8, §8. | Trade-off: fingerprints aren't comparable across stores — acceptable; cross-store comparison was never a feature. |
| F10 | Project-local history default risky (accidental commits) (High) | **A** | Default flipped to **user-local** `~/.prompt-debugger/stores/<project-key>/`. Project-local storage is explicit opt-in; when chosen, the store directory is created **self-ignoring** (a `.gitignore` containing `*` inside `.prompt-debugger/`) plus a README-in-store warning. §6.8, §8. Supersedes v0.1 open question #3. | |
| F11 | Storage lacks locking, migration, integrity, ordered ids (Medium) | **PA** | Accepted: cross-platform advisory locking (`fcntl`/`msvcrt` shim), atomic single-`write` appends on `O_APPEND` fds, per-record `schema_version` + `migrate` command (backup first), `doctor` command (quarantines corrupt lines to `history.rejects.jsonl`), `archive` rotation. **Partial on the letter**: UUIDv7 is not in Python 3.10 stdlib, so ids are `pd-<epoch-ms>-<uuid4[:8]>` — the same time-ordered-unique property without a dependency. §6.8. | |
| F12 | Network CI gate naive (grep) (Medium) | **A** | Policy restated precisely: **no runtime network I/O** in shipped code. Enforced by (a) an AST-based import allowlist per module (only named stdlib modules permitted — this also catches `subprocess`, which is not on the allowlist), and (b) a runtime test harness that replaces `socket.socket` with a raising stub for the entire unit-test suite. §7 S6, §11. | |
| F13 | Windows portability underdesigned (Medium) | **A** | Launcher strategy specified (`python3` → `python` → `py -3`, with all three patterns pre-approved in `allowed-tools`); no Bash-isms in documented commands; 3-OS CI matrix (ubuntu/macos/windows); fresh plugin-install smoke test moved from M5 to **M1 exit criteria**. §6.9, §11, §12. | The maintainer's own environment is Windows, which makes this finding directly testable from day one. |
| F14 | Testing underweights semantic quality (High) | **A** | Three new eval suites with documented protocols: **meaning-preservation** (paired before/after prompts, LLM-judge with human spot-check, snapshot regression), **red-team rewrite** (laundering attempts must be declined; gate outcomes asserted), **rubric calibration** (prompts with planted issues; detection precision/recall tracked per release). §10. | These are model-in-the-loop and run at release gates, not per-PR — cost-realistic for an OSS project. |
| F15 | CI/CD lacks release trust (Medium) | **PA** | Accepted: SHA-256 checksums for release artifacts, **GitHub artifact attestations** (provenance without maintainer key custody), pinned marketplace refs to tags, compatibility matrix (Claude Code min version / OS / Python), release security checklist. **Partial**: GPG-signed tags are *recommended, optional* — mandating personal key management for a community project raises the contribution bar for marginal benefit over attestations. §11. | |
| F16 | Threat model gaps: terminal escapes, symlink races, malicious history, export injection, update compromise, permission drift, injection-steered tool use (Review §7) | **A** | All added: control-character/ANSI sanitization on any echoed stored content (S8), CSV formula-injection escaping in exports (S9), stored records treated as untrusted input — validated and sanitized on read (S10), storage path resolution + containment + symlink refusal (S11), release trust (S12, via F15), CI permission-profile contract test (S13, via F4), `disallowed-tools` against injection-steered writes (S1/F4). §7. | |
| F17 | Privacy: raw storage should be rare/loud/reversible; exports redacted by default (Review §8) | **A** | Raw storage requires per-save interactive confirmation, records are flagged and visibly marked in `list` output, and `strip-raw <id>` converts raw→redacted in place. Exports are redacted and fingerprint-free by default; including raw text requires an explicit flag that prints a warning. §8. | |
| F18 | Performance thresholds asserted without benchmarks (Review §9) | **A** | Benchmark fixtures (synthetic 10k/100k-record stores) with committed benchmark script; measured thresholds replace the asserted 50k figure; defined max-size behavior (warn + suggest `archive`). §9. | |
| F19 | Docs: ADRs, glossary, data lifecycle, compatibility policy, unknown-event handling (Review §11–13) | **A** | `docs/adr/` (ADR-0001 core/adapter split, ADR-0002 dual-artifact reports, ADR-0003 schema subset validator, ADR-0004 storage default, ADR-0005 no browser extension), `docs/GLOSSARY.md`, data lifecycle section in PRIVACY.md, compatibility policy in README/INSTALL, unknown-event handling in taxonomy. §5, §12. | |

## Findings rejected

None rejected outright. Two partial acceptances (F1 runtime scope, F15 signed tags) and one letter-vs-spirit substitution (F11 id scheme) are justified above on engineering grounds (YAGNI on unproven adapters; key-custody burden vs. attestation equivalence; stdlib constraint).
