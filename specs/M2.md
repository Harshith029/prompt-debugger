> This document describes *what* M2 must deliver and *how success is judged*. It deliberately contains no code design beyond what the frozen architecture and contracts already record, and no analyzer or rewrite logic (M3). Every capability named here is reserved for M2 by [docs/ARCHITECTURE.md](../docs/ARCHITECTURE.md) (§§4, 6.7–6.9, 9, 10), the contracts under `core/contracts/`, the invariants catalogue ([docs/CONTRACT-INVARIANTS.md](../docs/CONTRACT-INVARIANTS.md)), or the obligations recorded at M1 close-out ([docs/releases/M1.md](../docs/releases/M1.md) §16); nothing is invented here.

## Background

M0 froze the contracts and invariants; M1 delivered the verified knowledge corpus and the declarative policy layer. At M2's start, the runtime library (`src/prompt_debugger/`) contained only version constants and path resolution — every deterministic behavior the product depends on (storage, validation, verification, redaction, sanitization, rendering, the CLI) existed only as contract prose, schemas, and reference tests. All twelve functional requirements have since been implemented, independently approved, and frozen: the schema-subset validator, the invariant verifier, redaction, sanitization, the storage layer, rendering, the knowledge accessors, the status-promotion pass, the ADR-0009 re-deferral, the CLI, the performance harness, and version bookkeeping. The milestone is closed and recorded in [docs/releases/M2.md](../docs/releases/M2.md). Invariants IR-1/2, RPT-1–4, EV-2, PR-2–6, PT-1–3, VC-2, and PL-6 are catalogued with enforcement point "M2" (PL-6's actual enforcement is deferred beyond M2 — see FR-2).

## Problem statement

Nothing user-facing can ship until the deterministic layer exists:

1. The analyzer (M3) cannot emit trustworthy reports until evidence quotes are machine-verified (IR-1/RPT-1) and reports are schema-validated by the runtime, not only by dev tooling.
2. Persistence cannot exist until the storage contract's write path (validate → redact → fingerprint → atomic append under lock) is implemented — the privacy invariant PR-1 is currently enforced only against fixtures.
3. The `history` skill (M4) has no CLI to dispatch to; its launcher honestly reports not-implemented.

M2 closes exactly this gap: it implements as code what M0/M1 recorded as contracts, invariants, and knowledge — and nothing more.

## Goals

- Implement the seven stdlib-only library modules the architecture reserves for M2: `schema`, `verify`, `redact`, `sanitize`, `store`, `render`, `cli`.
- Make every invariant currently marked "M2" in the invariants catalogue executable at runtime — **except PL-6**, whose enforcement requires a policy-executing engine that M2 does not build (see FR-2) — keeping the existing reference tests as the semantic baseline.
- Implement the Knowledge Engine query model the knowledge contract fixed at M0 ("implemented M2, contract now").
- Discharge the M1-inherited obligations: differential `schema.py`-vs-`jsonschema` validation; the status-promotion pass when the library first binds to entry statuses; the ADR-0009 revisit.
- Replace guessed performance constants with measured ones (store generators + timing harness).

## Non-goals

- Implementing analyzer or rewrite behavior, prompt-quality judgment, or any LLM-facing logic (M3).
- Wiring the `analyze`/`rewrite`/`history`/`pd` skills to the library, or changing skill prose (M3–M4).
- Building the semantic-evaluation harnesses (meaning preservation, red-team rewrite, rubric calibration — M3).
- A Prompt Tree renderer (post-v1 roadmap; M2's verifier only checks PT-1–3 on prompt-tree documents).
- New providers, new knowledge-pack types, or knowledge content changes beyond the promotion pass (FR-8), its bookkeeping, and the single explicitly scoped FR-9 exception below.
- Contract or schema changes — with **exactly one carved-out exception**: if the FR-9 review of ADR-0009 decides to add the truncation stop-condition kind, that decision authorizes one additive `kind` enum member in the Observable Event contract plus its taxonomy entry, tests, and snapshot bump, taken as an approved contract evolution under the frozen-baseline exception. If the review re-defers, scope stays fully closed and no contract or taxonomy change occurs in M2.
- Adding runtime dependencies or runtime network I/O (ADR-0006 — permanent).
- Packaging, installation documentation, or release-trust work (M5).

## User stories

- *As the M3 analyzer,* I need `verify` to reject a report whose evidence quote is not a verbatim substring of its reference prompt, so hallucinated evidence becomes a caught failure instead of an invisible one.
- *As a user saving an analysis,* I need the write path to redact the entire record — including the embedded report — so a secret can never survive persistence unless I explicitly, loudly opted into raw storage.
- *As a user on Windows, macOS, or Linux,* I need every storage operation (locking included) to behave identically, launcher-agnostically.
- *As a maintainer,* I need the in-repo subset validator proven equivalent to `jsonschema` on every schema and fixture, so owning a validator stays a bounded risk.
- *As a future adapter author,* I need read-only knowledge accessors so no consumer ever parses pack files ad hoc.

## Functional requirements

1. **Schema-subset validator (`schema`).** Validates any instance against the repository's restricted JSON-Schema subset (contracts README), including composite validation (CV-1/CV-2: report → ir/event/rewrite; history record → embedded report). Rejects schemas that use constructs outside the subset. CI gains **differential validation**: `schema.py` and the dev-only `jsonschema` package must agree on accept/reject for every repository schema and a fixture corpus containing both accept and reject cases; disagreement fails the build (ARCHITECTURE §6.7; M1 §16 obligation).
2. **Invariant verifier (`verify`).** Runtime enforcement of the catalogued relationships schemas cannot express: IR-1/IR-2, RPT-1–RPT-4, RW-1–RW-3, EV-1, EV-2 (against the loaded taxonomy version), PT-1–PT-3. Verification failures are returned as structured, machine-readable violations (id, path, message) — the one-repair-pass interaction loop belongs to the M3 skill workflow, not this module. The existing reference checkers in `tests/` remain and must agree with `verify` on the same fixtures. **PL-6 is explicitly out of scope:** it states that `rationale` fields never influence a policy decision, and M2 builds no policy decision engine, so there is no decision path to instrument; its runtime enforcement remains deferred until the milestone that introduces policy execution (M3+), and the invariants catalogue's enforcement column for PL-6 is updated by that milestone, not this one.
3. **Redaction (`redact`).** Deterministic, pattern-based secret/PII scrubbing with **typed placeholders**, applied to whole records per PR-1 (every content-bearing field of a `raw: false` record, including the embedded report). Ships with a true-positive/false-positive corpus as unit tests, and documents its honest limits consistently with [docs/PRIVACY.md](../docs/PRIVACY.md).
4. **Sanitization (`sanitize`).** Strips C0/C1 control characters and ANSI CSI sequences from any content printed or exported (threat S8); prefix-escapes CSV cells with leading `= + - @` or tab (S9). Applied on every read/display/export path (records are untrusted input, S10).
5. **Storage (`store`).** The storage contract's layout, config, and operations, restricted to what the frozen contracts make objectively implementable: on-disk layout and per-store salt; config (schema-validated; project-local opt-in requires `storage_scope: project` **and** `project_local_acknowledged: true`, created self-ignoring per ADR-0004); operations `append` (validate → redact → fingerprint → single atomic `O_APPEND` write under the store lock; refuses unvalidated payloads; refuses a `raw: true` payload absent explicit per-save confirmation), `list`, `get`, `compare`, `trends`, `export` (redacted and fingerprint-free by default; raw requires an explicit warning flag), `delete`, `purge`, `strip-raw`, `doctor` (quarantine to `history.rejects.jsonl` with reasons), `migrate` (timestamped backup first; accepts every `record_version` ever shipped, PR-6), `archive`. Enforces PR-2 (id format/ordering), PR-3 (HMAC-SHA256 fingerprints, store-local, excluded from default exports), PR-4 (atomic appends; readers tolerate and report a trailing partial line), PR-5 (symlink refusal + path containment, S4), and PR-1 at write time. Locking via the documented `fcntl`/`msvcrt.locking` shim.
    **`compare` scope decision (explicit):** the storage contract describes `compare` as "rubric-score delta + unified diff of redacted prompts", but no frozen contract defines a rubric *score* or any aggregation — the Report schema defines only per-finding `dimension` and `severity`. M2 therefore implements only the contract-grounded portion: the **unified diff of redacted prompts**. **Rubric-score-delta behavior is deferred** until an approved contract evolution defines scoring; M2 must not invent a scoring formula or any replacement output.
6. **Rendering (`render`).** Report JSON → Markdown projection (ADR-0002: the Markdown a user reads is regenerable from the canonical report) and md/csv/json exports with provenance headers. All fixed notice wording is emitted from `notices.json` (the single source of truth) — never re-authored in code.
7. **Knowledge accessors.** The read-only query model fixed in the knowledge contract: `load_manifest()`, `load_pack(provider)`, `technique(id)`, `rubric_dimension(id)`, `event_entry(id)`, `claims(status=...)`. Accessors validate what they load (via `schema`) and never mutate packs.
8. **Status promotion (inherited from M1), narrowed to claim-grounded artifact classes.** With FR-7 the library becomes the first consumer bound to entry statuses. Promote `draft → active` those **techniques and event-taxonomy entries** whose support holds — every cited claim `verified` — because those are the only artifact classes with a claim-provenance relation (`source_claims`, governed by KN-2). **Rubric dimensions and patterns are excluded:** they carry `status` fields but no claim-support relation (they reference techniques/dimensions, not claims), so no existing repository artifact grounds a promotion criterion for them; defining one (for example, "active when all referenced techniques are active") would invent new metadata semantics. They remain `draft`, and their promotion criterion is recorded as open choice O7 for an explicit governance decision — not silently decided here. Promotion of the grounded classes bumps the knowledge snapshot label per KN-10 and is recorded in the changelog; it is a reviewed content change following [docs/process/policy-review.md](../docs/process/policy-review.md) where it touches policy-adjacent files.
9. **ADR-0009 revisit (inherited from M1).** Explicitly decide the truncation observable, under the single carved-out exception stated in Non-goals: **either** add the stop-condition kind — one additive `kind` enum member in the Observable Event contract plus its taxonomy entry, tests, and snapshot bump, executed only after explicit approval as a contract evolution under the frozen-baseline exception — **or** re-defer with reasons in a superseding ADR, in which case no contract, schema, or taxonomy change occurs in M2. The decision must be recorded either way; silence is not an outcome. This FR is the *only* path by which M2 may touch a frozen contract or the taxonomy, and it is bounded to exactly the change described.
10. **CLI (`cli`).** An argparse entry point exposing the storage-contract operations plus `verify`/`render` where user-invocable, launcher-agnostic (`python3` / `python` / `py -3`), the dispatch target the M4 `history` skill shim will call. No skill files change in M2.
11. **Performance measurement.** Synthetic store generators (10k / 100k records) and a timing harness under `benchmarks/perf/`, run on the CI matrix; the store size-warning threshold is set from these measurements, not guessed (ARCHITECTURE §9). Results are recorded in the milestone record.
12. **Version bookkeeping.** Library/adapter versions bumped per the release workflow; knowledge snapshot bumped if and only if knowledge content changed (FR-8 does); every interface addition recorded in the changelog; `CONTRACT_VERSIONS` stays consistent (VC-4).

## Non-functional requirements

- **Stdlib-only, no runtime network I/O** (ADR-0006): enforced by the existing AST import allowlist and socket-block harness; new modules join the allowlist review.
- **Determinism:** identical inputs produce identical outputs on all three OSes; no wall-clock, locale, or ordering dependence in outputs (record ids embed time by design and are the only sanctioned time-dependent value).
- **Typed and strict:** `mypy --strict` clean; public functions fully annotated.
- **Host-neutral:** the library never references a host product; adapters remain the only host-specific layer (ADR-0003).
- **Honest failure:** every rejection (validation, verification, lock contention, migration) produces a specific, actionable error — never silent fallback.

## Constraints

- M0 contracts and M1 knowledge/policy are frozen; M2 implements them and may change them only through the objective-defect path or the explicitly scoped FR-9 decision.
- The schema subset is fixed; `schema.py` implements exactly it (no extension, no laxity).
- The declarative policy layer stays declarative: the library reads policy data; nothing in M2 interprets or executes policy semantics (that is the M3 analyzer's job) beyond loading and validating the files.
- Python ≥ 3.10, three-OS CI matrix, `ruff==0.15.21` pinned formatter discipline (bump = pin update + reformat in one change).

## Architecture impact

None structural: M2 fills the `src/prompt_debugger/` layer the architecture already draws and adds `benchmarks/perf/`. Contracts, schemas, and the taxonomy change **only** if the FR-9 review decides to add the truncation kind — the one bounded exception carved out in Non-goals; if FR-9 re-defers, they do not change at all. Adapters, skills, and the knowledge corpus (beyond FR-8's narrowed promotion and FR-9's possible taxonomy entry) are unchanged.

## Deliverables

- The seven library modules, fully typed, with unit tests per module (including the redaction true/false-positive corpus and storage concurrency/corruption tests).
- Differential-validation CI (subset validator vs `jsonschema`).
- Runtime enforcement of every invariant currently marked "M2" — **excluding PL-6**, which remains deferred until the milestone that introduces policy execution (M3+) — with the invariants catalogue updated to name the enforcing module and test.
- Knowledge accessors; the promotion pass with its snapshot bump; the ADR-0009 decision record.
- `benchmarks/perf/` with recorded measurements and the measured size-warning threshold.
- CLI ready for M4 skill wiring; changelog entries; a milestone record `docs/releases/M2.md` at close-out.

## Acceptance criteria

- Every invariant listed in [docs/CONTRACT-INVARIANTS.md](../docs/CONTRACT-INVARIANTS.md) with enforcement point "M2" — **excluding PL-6**, which is deferred to the milestone that introduces policy execution — is enforced by library code and covered by at least one positive and one negative test; the catalogue's enforcement column is updated accordingly.
- Differential validation runs in CI over all repository schemas plus an accept/reject fixture corpus, and passes.
- The PR-1 privacy tests pass against the **real write path** (not only fixtures): a leak fixture appended through `store.append` is redacted in every field.
- **Raw persistence is rare, loud, and gated:** an `append` carrying `raw: true` **without** explicit per-save confirmation is rejected — nothing is written and a specific error is returned; an `append` carrying `raw: true` **with** explicit per-save confirmation succeeds, the stored record carries the visible `raw` flag surfaced by `list`, and `strip-raw` converts it to redacted in place. Confirmation is per save — it is never a persisted default, and no configuration can make raw storage silent. Both behaviors are covered by tests.
- `store` operations behave per the storage contract on all three OSes, including lock contention, partial trailing lines, doctor quarantine, and migrate-with-backup; `compare` produces the unified diff of redacted prompts per the FR-5 scope decision — no score and no substitute aggregate is computed or displayed.
- All **techniques and event-taxonomy entries** whose cited claims are all `verified` are `active`; rubric dimensions and patterns remain `draft` with the deferral recorded; the snapshot label is bumped and KN-2/KN-7/KN-10 tests pass on the promoted corpus (FR-8) — or a reviewed record explains any exception.
- The ADR-0009 decision exists (new kind implemented and tested under the carved-out exception, or a superseding re-deferral ADR with no contract/taxonomy change).
- No new runtime dependency; the import allowlist and socket-block harness pass unchanged.
- Every M0/M1 quality gate remains green; no M1 artifact modified except through FR-8/FR-9 or an objective defect.

## Quality gates

`ruff check` · `ruff format --check` (pinned) · `mypy --strict` · import policy (AST allowlist + socket block) · version consistency · schema validation (subset + instances + **differential**) · link check · plugin vendor sync · benchmark validation (corpus + perf harness) · full pytest on the 3-OS × 2-Python CI matrix. All must pass at every FR boundary, not only at close-out.

## Test strategy

- Unit tests per module as reserved in ARCHITECTURE §10 (store locking/atomic-append/migrate/doctor/archive; schema; verify; redact corpus; sanitize; render; cli).
- The existing reference checkers stay authoritative: `verify` must agree with them on the shared fixtures (same accept/reject verdicts), preventing semantic drift between M0's frozen rules and M2's implementation.
- Storage tests include concurrency (two writers under the lock), corruption (trailing partial line, hand-crafted invalid records → doctor quarantine), and symlink/containment attacks (S4, S10).
- Differential validation fixtures include deliberate out-of-subset schemas (must be rejected by the meta-check) and instance accept/reject pairs per contract.
- Promotion (FR-8) is verified by the existing KN-2/KN-7/KN-10 integrity tests running against the promoted corpus.

## Risks

- **Owning a validator.** Bounded exactly as designed at M0: restricted subset + differential testing against `jsonschema` (FR-1); any disagreement is a build failure, not a judgment call.
- **Cross-platform locking and file semantics.** Mitigated by the documented shim, CI on all three OSes, and concurrency tests; Windows-specific behaviors (locking ranges, path resolution) get dedicated cases.
- **Redaction overreach or underreach.** Mitigated by the true/false-positive corpus and by PRIVACY.md's honest-limits documentation; redaction failures must fail closed (refuse the save) rather than persist unredacted content silently.
- **Scope creep into M3.** The analyzer temptation is explicit: `verify` returns violations, it does not repair; `render` projects reports, it does not author them. The non-goals bind.

## Deferred work

- **Rubric-score semantics** (a defined score/aggregation for reports and `compare`'s score-delta) — requires an approved contract evolution; until then no component computes or displays a score.
- **Promotion criteria for rubric dimensions and patterns** — no claim-provenance relation exists for these classes; defining one is a governance decision (O7), not an M2 deliverable.
- **PL-6 runtime enforcement** ("rationale never influences a decision") — deferred until the milestone that introduces policy execution (M3+); M2 has no policy decision engine to instrument.
- Analyzer/rewrite skills, trigger/adversarial/meaning-preservation/rubric-calibration evals — M3.
- `history` skill wiring and privacy UX verifications (raw-storage confirmation flow, export warnings in situ) — M4.
- INSTALL/USAGE documentation, packaging, release-trust pipeline, 3-OS install smoke tests — M5.
- Prompt Tree renderer; MCP/CLI host adapters — post-v1 roadmap.
- Any knowledge re-verification beyond FR-8 — the quarterly re-verification cadence continues independently.

## Out of scope

Everything in Non-goals and Deferred work; UI of any kind; telemetry (permanently); modification of the frozen policy semantics.

## Definition of done

M2 is done when the acceptance criteria are met, all quality gates pass on the CI matrix, an independent verification has approved the milestone, `docs/releases/M2.md` records it (including the FR-8 promotion decision, the FR-9 outcome, and the measured performance numbers), and the library is frozen as the deterministic base on which M3 builds the analyzer.

## Transition to M3

M2 hands M3 a deterministic substrate: schema-validated contracts at runtime, machine-verified evidence, privacy-enforcing storage behind a CLI, rendering from the canonical report, and a promoted, versioned knowledge corpus behind read-only accessors. M3 then authors the `analyze`/`rewrite` skill behavior against those accessors and the policy data — the first milestone in which model judgment enters the system, constrained on every side by what M0 specified, M1 verified, and M2 enforced.
